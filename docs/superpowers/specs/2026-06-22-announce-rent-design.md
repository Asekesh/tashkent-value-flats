# Анонс аренды пользователям бота

Дата: 2026-06-22
Статус: согласован

## Контекст

Шаг 5 («бот учится аренде») в проде (commit 26facf6): в `/new` появился тумблер
Продажа/Аренда. Существующие пользователи об этом не знают. Нужно сообщить.

Выбранный канал (решение пользователя): **разовая рассылка всем активным
пользователям + правка welcome/help** для новичков.

Наружное слово — **«уведомление»** (как уже во всех кнопках бота:
«➕ Новое уведомление», «📋 Мои уведомления»). НЕ «алёрт». Внутренние имена в
коде остаются alert.

## Решение

**Разовый скрипт**, а не HTTP-эндпоинт. Причина: роутер `/api/admin` без auth —
открытый эндпоинт, рассылающий всем, опасен (спам при случайном/повторном
вызове). Скрипт запускается вручную через `railway run` (привычный паттерн, как
бэкфилл ЖК) — нулевой публичный surface, естественно run-once.

## Компоненты

### 1. Скрипт `backend/app/scripts/announce_rent.py`

Запуск:
```
railway run -s Postgres python -m app.scripts.announce_rent          # боевой
railway run -s Postgres python -m app.scripts.announce_rent --dry-run # только счёт
railway run -s Postgres python -m app.scripts.announce_rent --after-id 1234  # резюм
```

Поведение:
- Выбирает пользователей: `telegram_id IS NOT NULL AND is_active`, сортировка по
  `id ASC`, `id > after_id`.
- Для каждого: локализованный текст (`user.lang` → ru/uz) + inline-кнопка
  «➕ Новое уведомление» с `callback_data="start:new"` (ведёт на тумблер
  Продажа/Аренда — обработчик `start_new` уже есть).
- Отправка через `https://api.telegram.org/bot{token}/sendMessage` синхронным
  httpx (как `notifier._send_listing_sync`), `parse_mode=HTML`,
  `disable_web_page_preview=True`. Токен из `settings.telegram_bot_token`.
- Throttle: `time.sleep(0.05)` между отправками (~20/сек, под лимит Telegram ~30).
- `429` → `time.sleep(retry_after + 1)`, повтор той же отправки один раз.
- `403` (бот заблокирован) → пометить `user.is_active = False` (как делает
  notifier), считать как blocked, идти дальше.
- Прочие ошибки сети — логировать, считать failed, НЕ падать.
- Прогресс: печатать `sent/total`, last_id каждые 50 и в конце — итог
  `sent / blocked / failed`, последний обработанный id (для резюма).
- Флаги: `--dry-run` (посчитать получателей, ничего не слать), `--limit N`,
  `--after-id N`.

**Курсорный резюм (`--after-id`), а не limit-голова** — урок бэкфилла ЖК: при
обрыве повторный прогон с limit без курсора начал бы слать с начала (двойная
рассылка). Скрипт печатает last_id, чтобы продолжить с него.

Идемпотентность опирается на ручной запуск + курсор; отдельной БД-метки нет
(проще; цена ошибки — один лишний прогон, гасится передачей `--after-id`).

### 2. i18n (`backend/app/bot/i18n.py`)

Новый ключ `announce_rent` (ru/uz):
- ru: «🔑 <b>Теперь есть уведомления об аренде!</b>\n\nБот ищет квартиры не только
  на продажу, но и в аренду — можно с фильтром «без комиссии».\n\nНажми
  «➕ Новое уведомление» и выбери «Снять» на первом шаге.»
- uz: аналог.

Кнопка использует существующий ключ `b_new` («➕ Новое уведомление» /
«➕ Yangi bildirishnoma»).

### 3. Welcome / help (`backend/app/bot/i18n.py`)

В ключах `welcome` и `help` упомянуть, что бот следит за квартирами **на продажу
и в аренду** (правка текста, без новых шагов). Сохранить существующий тон/длину.

## Вне скоупа

- Веб-баннер на сайте (фронт переделывается — [[project_frontend_redo_krisha]]).
- Generic broadcast-эндпоинт/UI в админке (YAGNI; нужен один разовый анонс).
- БД-метка идемпотентности (курсора достаточно).

## Критерии готовности

- Скрипт шлёт корректный локализованный анонс с рабочей кнопкой «➕ Новое
  уведомление» (открывает тумблер Продажа/Аренда).
- `--dry-run` печатает число получателей, ничего не отправляя.
- 403 деактивирует юзера и не роняет прогон; 429 ретраит.
- `--after-id` позволяет продолжить с места обрыва.
- welcome/help упоминают аренду; слово «уведомление», не «алёрт».
